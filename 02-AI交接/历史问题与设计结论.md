# 历史问题与已经确定的设计结论

这份文档防止后续维护者重复走已经失败的路线。

## 1. 触摸偏移不是单一固定像素问题

曾出现：顶部死区、普通模式勉强准确但全屏失准、不同网页比例不同。根因是手机 CSS 像素、JPEG 像素、screencast DIP、页面 CSS 像素、`pageScaleFactor` 和异步布局指标被混用。

结论：触摸必须绑定生成当前画面的同一帧元数据，使用归一化坐标；一次手势中冻结标签、帧代际、视口修订和校准参数。固定偏移可校准，比例只在边缘渐进误差时调整。

## 2. 校准不能依赖目标网页 DOM

向 ChatGPT／Claude 页面插入红点或校准元素既会干扰页面，也不利于严格人工模式。

结论：所有校准标记、蓝色实际触点和红色命中点只画在手机控制页面的本地覆盖层。

## 3. 静止网页没有新帧不等于黑屏

反复停止／启动 screencast 会制造闪烁和黑屏。

结论：只有在用户操作后需要更新且长时间无帧时，才逐级截图、重启 screencast、重附加和重连；始终保留最后成功画面。

## 4. 手机处理不过来不能排队所有帧

旧帧积压会让手机落后数秒。

结论：使用 frame ACK；上一帧未显示时只保存最新待发帧。

## 5. `visibilityState=visible` 不能代表电脑当前标签

电脑切到其他应用时，Edge 页面仍可能 visible，曾被误判为切换标签并导致重附加卡住。

结论：v6.7 默认使用 Windows UI Automation 确认前台进程和选中标签；不是 Edge 时保持当前标签。严格模式不允许自动回退到 Runtime 焦点轮询。

## 6. 文件上传不能长期拦截

长期 `Page.setInterceptFileChooserDialog(enabled=true)` 会让普通浏览会话持续处于不必要的武装状态。

结论：先点控制器“上传”，再点网页上传；完成、取消、超时、换页和断线后立即解除。

## 7. ChatGPT／Claude 不适合长期完整手机设备仿真

桌面 Edge 身份加手机 screen、触摸点和 coarse pointer 会形成无必要的混合环境。

结论：严格人工模式保留真实桌面 Edge 身份，使用真实窄窗口和鼠标／滚轮；原生触摸只由用户临时启用。不得通过 UA、webdriver 或指纹伪装来“修复”不一致。

## 8. 代理和原配置不是排查变量

用户已经明确：固定代理质量已经长期验证，原 Edge 状态必须继承。

结论：任何后续 AI 不得建议取消代理、直连、自动换节点或改成隔离浏览器作为默认解决方案。

## 9. 交付不能再发零散文件

历史上用户多次收到开发候选或单个核心文件，无法直接使用。

结论：每次交付都必须是完整 ZIP、SHA-256、更新说明和验证报告；若尚未完成，必须明确是开发候选，不能冒充正式发布。v6.8.0 起打包由 CI 自动完成、仓库为唯一事实源，但“完整交付”的要求不变。

## 10. 体验优化必须落在“网页 JS 观测不到”的层面（v6.8.0）

v6.8.0 讨论过多种提升阅读/操作体验的方案。判断标准最终统一为一句话：**这个优化会不会被目标网页的 JavaScript 观测到，会不会需要激活窗口或注入内容。**

- 静止画面高清化选择 `Page.captureScreenshot`（合成器层）而不是任何页面内重排/放大，正是因为截图对网页不可见；并且明确规定补拍**不激活窗口**——因为 `Target.activateTarget`/`Page.bringToFront` 会触发页面可见的 `focus`/`visibilitychange`，那才是真正的可疑信号。
- 剪贴板桥选择 PowerShell `Get/Set-Clipboard`（操作系统层）而不是 `navigator.clipboard` 注入或页面脚本读取，因为前者完全在浏览器之外，网站看不到。
- 手机专用窗口用同一 profile 新开普通窗口，而不是任何指纹层面的花招，因为它对平台就是“同账号又开了一个窗口”。

结论：任何新体验功能，先问这三点；凡是能在 OS 层或合成器层完成的，绝不放进目标网页。这条已写入规范第 3、10 节。

## 11. 静止补拍不能变成轮询，也不能循环

早期设想里，“定时刷新清晰截图”很容易滑向“每隔 N 秒截一次”甚至“截图后再截图”。

结论：补拍必须由“持续无新帧且无输入”这一真实静止条件触发一次；补拍帧带 `source=idle-sharpen`，其本身不再触发补拍；有任何输入或新 screencast 帧立即回到实时流。它是锦上添花，失败只记录、不进入重启/重连恢复流程，避免为一张清晰图片扰动画面通道。

## 12. 明确永久否决的方向（防止后续 AI 以“提升体验/清晰度”重新提出）

以下方向已被反复评估并永久否决，不是“尚未实现”，后续 AI 不得以体验为名重新引入：

- **DOM 镜像渲染**（把 ChatGPT/Claude 页面结构传到手机重新排版显示）：必然需要读取并解析目标页面 DOM，直接触碰 HUMAN-003 / STRICT-005 / STRICT-007 红线。清晰度问题由静止 PNG 补拍解决，不需要动 DOM。
- **为观感激活/置顶窗口**：任何纯显示目的的 `Target.activateTarget`/`Page.bringToFront` 都会给页面可见的 focus/visibilitychange 信号，得不偿失。
- **自建 WebRTC / 视频编码通道**：CDP 无视频编码接口，自建等于重写 Sunshine/Moonlight。若将来对帧率不满，正确做法是先用现成方案对同一 Edge 窗口做 A/B 实验用数据决定，而不是在本项目里写编码器。
- **系统音频转发**：CDP 不传音频；需要语音只能走窗口捕获类方案，那是另一个项目。
- **反检测 / stealth / 指纹伪装 / 随机人类行为 / webdriver 隐藏**：与项目定位根本冲突（见第 7 节和规范 5.6）。
