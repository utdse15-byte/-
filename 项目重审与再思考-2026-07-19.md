# 项目重审与概念再思考（针对 v6.7.0，2026-07-19）

本文是一次从零出发的整体评审：先回到目标本身，盘点方案空间，再给出具体建议。
本文只是评审意见，未改动任何代码；涉及规范的建议均已标注需求 ID，MUST 项无任何静默变更。

---

## 一、我理解的你的真实目标

从全部交接文档反推，这个项目的目标可以压缩成一句话：

> **随时随地以"电脑上那个长期稳定的 Edge 会话"使用 ChatGPT / Claude** —— 同一登录、同一代理出口、同一浏览器环境、纯真人操作，把账号侧的风险变量压到最少；同时手机端体验尽量接近原生手机浏览器。

三条底层约束支撑它：

1. 固定代理 `127.0.0.1:7897` 是长期验证过的基础设施，不是变量；
2. 绝不做自动化、抓取、批量、伪装 —— 项目的合法性和低风险恰恰来自"它只是遥控器"；
3. 交付必须完整、可复现、可交接给下一个 AI。

下面所有建议都在这三条约束之内。

---

## 二、从零重想：方案空间盘点

如果今天从一张白纸开始设计，可选路线有五条：

| 路线 | 描述 | 判断 |
| --- | --- | --- |
| A | CDP screencast 遥控真实 Edge（现方案） | **正确选择**，理由见下 |
| B | 单窗口硬件视频流（Sunshine/Moonlight 类）+ 轻量伴随控制器 | 画质/帧率/音频全面占优，但丢失网页级语义；只值得做实验，不值得重写 |
| C | 全桌面远控 | 你已否决，同意：手机上操作整个桌面的体验确实差 |
| D | 手机原生浏览器 + 家庭网络出口（如 Tailscale exit node） | 出口 IP 相同但**会话/设备/指纹全变**，恰好制造你要避免的"异常登录"信号，否决 |
| E | A 为控制平面 + B 为可选视频通道的混合 | 未来若帧率成为硬瓶颈时的演进方向，现在不做 |

**为什么 A 是对的**：路线 B/C 解决的是"像素搬运"，而这个项目的真正价值在"网页级语义"——标签跟随、按需文件上传、最小页面介入、严格人工模式的可解释性。这些只有站在浏览器协议层才能做到。路线 D 则从根上违背"同一会话"目标。

**诚实的整体评价**：这个项目最难得的不是代码，而是需求工程。规范 / 基线 YAML / 实现对照表 / 历史问题与设计结论 / 交接提示词这一整套体系，在个人项目里极其罕见，正是 AI 接力开发的正确姿势。核心架构决策——真实窄窗口而非设备仿真、UIA 而非页面脚本轮询、歧义时保守不切、frameAck 最新帧优先、校准只画本地覆盖层——每一条都能在《历史问题与设计结论》里找到伤疤支撑。**概念本身不需要推倒重来**；值得做的是下面这些增量升级。

## 三、值得认真考虑的改进（按价值排序）

### 1. 用 Tailscale 把"局域网工具"升级为"随身工具"（高价值 / 低工作量）

现状的两个真实短板：

- 控制器只在局域网可用，**出门即失效**——而"在外面用手机访问电脑上的会话"恰恰是这个工具最自然的使用场景；
- 手机与控制器之间是**明文 HTTP**，同一 LAN 内的其他设备理论上可嗅探到令牌和画面。

建议：电脑与手机各装 Tailscale（WireGuard 私有组网，不开公网端口），控制器继续监听现端口，Windows 防火墙在 LocalSubnet 之外额外放行 Tailscale 虚拟网卡。这**不是**把端口映射到公网，SEC-002 的精神（不暴露公网、单一控制者、令牌）完全保留，传输反而从明文升级为端到端加密。

额外红利：`tailscale cert` 可以为本机拿到真实受信的 HTTPS 证书。手机端一旦变成安全上下文（HTTPS），就解锁三样东西：

- **PWA 正式安装**（现在 manifest.json 在明文 HTTP 下装不完整）；
- **Screen Wake Lock**——读长回答时手机不熄屏，日常体验提升明显；
- 剪贴板 API 等只在安全上下文可用的能力（配合建议 3）。

与硬约束的关系：Tailscale 只影响"手机 ↔ 控制器"这一段；Edge 上网仍然只走固定代理，NET-001 完全不受影响。SEC-002 需要一条 v1.2 增补（"允许私有加密组网接口"），建议先改规范再实施。

### 2. 静止画面高清化 / idle sharpening（高价值 / 少量代码）

现状：screencast 和截图兜底全部是 JPEG（quality 72，见 `server.js` 的 `Page.startScreencast` 与 `Page.captureScreenshot` 调用），读长文时小字发糊。而这个工具 90% 的使用时间是在**读静止的对话文本**——恰好是 JPEG 最吃亏的场景。

建议：页面静止 N 毫秒（无新帧、无手势、无按下状态）后，自动补拍一张 PNG（或 quality 95）截图替换最后一帧；一有输入立刻回到实时 JPEG 流。阅读时近乎无损，滚动时保持流畅。

这属于规范 2.3 明确允许的"画面编码"类基础设施自动化，对应 UX-001 的扩展，无 MUST 冲突。这是所有建议里"每天体感提升 ÷ 代码量"比值最高的一条。

### 3. Windows 剪贴板桥（高价值 / 需要一次规范讨论）

真实痛点：在手机上让 ChatGPT/Claude 写了一段文字或代码，**没有任何办法把它弄进手机的其他 App**；反向（手机长文粘进聊天框）已由输入面板 + `Input.insertText` 解决了一半。

建议的实现方式刻意绕开所有红线：参照 `helpers/edge-uia-monitor.ps1` 的模式，增加一个 PowerShell 剪贴板 helper（`Get-Clipboard` / `Set-Clipboard`）：

1. 用户在网页里**自己**点"复制"按钮（网页原生行为，内容进入 Windows 剪贴板）；
2. 用户在控制器界面点"取回电脑剪贴板"——显式、单次、不落盘、不解析。

全程不碰目标页 DOM、不用 Runtime，与 HUMAN-003 禁止的"自动读取/抓取/批量导出"性质不同——它等价于人自己把内容抄一遍，主语始终是用户的当次动作。

规范处理：建议作为 v1.2 新增条目（例如 CLIP-001：剪贴板传输必须由用户当次显式触发，单次、双向、不持久化、不解析内容、不自动同步），**先写规范再写码**，符合交接规则第 1 条。这一条请你先拍板，不拍板不做。

### 4. "专用窄窗口"选项，不再改动用户主窗口（中价值）

现状：严格模式把用户正在用的真实 Edge 窗口调成 560×960，污染桌面布局，退出后只能"尽力恢复"（已知限制 3）。

建议：增加一个**可选**模式——控制器在同一 profile 下新开一个独立 Edge 窗口（同进程、同会话、同 Cookie、同扩展），专供手机操作；原主窗口一动不动，退出即关窗，"窗口恢复失败"这一类问题整体消失。副作用是 UIA 跟随在该模式下可以锁定到这个专用窗口的 HWND，歧义反而减少。

代价："电脑切标签、手机跟随"的语义要按窗口区分（跟随专用窗口内的标签，忽略主窗口）。作为可选项提供、不改默认行为，PROF-001 / STRICT-002 不受影响。

### 5. 交付流程：从"ZIP 接力"升级为"Git + CI 接力"（对项目寿命最重要）

这套"完整 ZIP + SHA-256 + 人肉同步交接文档"体系，是在**没有代码仓库的年代**长出来的，它当时是对的。现在项目已经在 GitHub 上，但仓库只有一个 initial commit——版本历史仍然全靠文档口述。建议：

- **仓库成为唯一事实源**。现在 `01-程序/.../AI交接/` 和 `02-AI交接/` 是两份人肉同步的副本，已经存在漂移风险；改为仓库里只留一份，发布 ZIP 由 CI 自动打包时复制进去。
- **GitHub Actions 的 `windows-latest` 运行器自带 Edge Stable 和 PowerShell**。这意味着你路线图 P2 里的"Windows CI"不需要自建虚拟机：`edge-uia-monitor.ps1` 的真实解析、Edge 冒烟测试、甚至部分 UIA 控件树验证，可以在每次提交时自动跑。目前实现对照表里一长串"已实现/需 Windows 验证"欠账，有相当一部分能变成自动回归。
- `test/browser-ui.test.js` 在你的构建容器里被 URLBlocklist 策略跳过——在 GitHub CI 容器里没有这个策略，可以恢复执行。
- 每个正式版本打 **git tag**，ZIP 和 SHA-256 由 CI 生成并发布到 GitHub Releases。REL-003 的"完整交付"照旧满足，只是产线自动化、不可能再漏。
- 顺带：仓库名当前是 `-`，建议改成有意义的名字（如 `edge-phone-controller`）。

### 6. 安全细节收尾（小工作量）

整体安全设计已经不错（`timingSafeEqual` 比对令牌、CSP 与安全响应头齐全、上传文件名处理了 Windows 保留名、依赖只有 `ws` 一个包——供应链非常干净）。剩几处小尾巴：

- WebSocket 连接 URL 用 `?token=` 查询串携带令牌（`public/app.js:539`），HTTP API 已经用 Bearer 头了；建议 WS 改为连接后首消息认证（或 `Sec-WebSocket-Protocol`），让令牌彻底离开 URL，避免出现在任何日志里。
- 提供**一键轮换令牌**（现在需要手动删 `data/access-token.txt` 重启）。
- `/health` 无需令牌即返回版本、PID、连接数等信息；LAN 内可接受，但建议瘦身到 `ok` 一个字段，或同样要求令牌。
- 明文传输的根治靠建议 1。

### 7. UIA 的下一步：维持保守，增加可观测性

你路线图 P0 的"UIA 诊断快照"应该排在所有 UIA 兼容工作之前：每次"保持不切"都要能回答**为什么**（歧义 / 非前台 / 树不可用 / 地址读取失败），否则真机调试全靠猜。垂直标签、标签组、多语言按 P1 推进即可。

一条提醒：不要在任何体验压力下放宽"唯一匹配才切换"。同标题多会话是 ChatGPT/Claude 的常态，误切的代价（打断正在进行的对话、可能造成误操作）远大于手机上手动点一次标签。现在的保守设计是对的。

### 8. 想清楚就好、明确不建议做的

- **自建 WebRTC 视频通道**：CDP 没有视频编码接口，自建等于重写 Sunshine。若未来对帧率不满，正确做法是先拿 Sunshine/Moonlight 对同一个 Edge 窗口做一周 A/B 实验，用数据决定，而不是写码。
- **DOM 镜像渲染**（把页面结构传到手机重新排版）：直接触碰 HUMAN-003 / STRICT-005 红线，建议把"永久否决"写进《历史问题与设计结论》，防止后续 AI 以"提升清晰度"为名重新提出。
- **音频**：CDP 不传音频。若未来需要语音模式，唯一现实路径是窗口捕获类方案（路线 B），那是另一个项目。

## 四、代码与工程观感（简短）

- **亮点**：单一运行时依赖（`ws`）；令牌恒时比对；日志轮转与条数上限；诊断脱敏；测试分层诚实（QA-001 落实得很好——"最终压缩包验证结果"里明确列出未执行项，这种诚实在 AI 生成的项目里稀缺且宝贵）。
- **隐忧**：`server.js` 4673 行、`public/app.js` 4093 行的单文件已到临界——AI 一次上下文还能改，人类已经难以审阅。建议在下一个功能版本**之前**做一次纯结构拆分（CDP 会话、严格模式、screencast、上传、历史各成模块），行为零变更，靠现有测试网护航。`lib/` 的拆分先例已经存在，照做即可。
- **文档双份漂移**：见建议 5，这是当前交接体系里唯一的结构性隐患。

## 五、如果只做三件事

1. **Tailscale 接入 + HTTPS**（建议 1）——把工具从"在家可用"变成"随身可用"，顺带根治明文传输，这是对"你的目标"本身最大的放大器；
2. **静止画面高清化**（建议 2）——每天读长回答的体验立竿见影，代码量最小；
3. **Git + CI 交付流水线**（建议 5）——把"需 Windows 真机验证"的欠账变成每次提交的自动回归，项目才能长寿。

其中 1 和 3 涉及规范增补（SEC-002 扩展、REL-003 产线化），按你自己定的规则：先确认、再动手。
